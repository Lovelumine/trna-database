# ENSURE 图片资源管理目标方案

## 1. 文档目的

本文档定义 ENSURE 数据库图片资源管理的目标状态、实施边界、阶段路线和验收标准。

后续图片资源相关改造应以本文档为唯一基线，避免继续出现：

- 业务表图片字段、媒体库、前端硬编码 URL 三套体系并存
- 前端页面各自拼接图片地址
- 后台可以上传图片，但无法真正接管业务图片
- 删除图片时无法准确判断是否仍被引用
- 表格图片渲染规则散落在前端代码中，且已与真实数据库漂移


## 2. 当前现状

### 2.1 MySQL 现状

当前数据库已经具备媒体资产表，但尚未成为真实业务图像资源中心。

已确认事实：

- `media_assets` 当前记录数为 `0`
- `app_settings.table_media_field_config_json` 当前值为 `{}` 
- 数据库中图片资源仍主要依赖业务表里的 `pictureid`
- 文档图片主要仍在 Markdown 中直接写远程 URL

当前已知的业务图片型字段使用模式：

- `nonsense_sup_rna.pictureid`
- `frameshift_sup_trna.pictureid`
- 其他部分页面或功能仍通过 `pictureid -> MinIO URL` 规则拼图

当前问题：

- 媒体库表存在，但没有接管历史图片
- 没有“图片和业务实体/页面/文档之间的绑定关系表”
- 没有统一的结构化引用记录


### 2.2 后端现状

当前后端已经有媒体库基础接口：

- 媒体资产模型：`Flask/app/models.py`
- 媒体列表接口：`/admin/api/media`
- 媒体上传接口：`/admin/api/media/upload`
- 媒体删除接口：`/admin/api/media/<id>`
- 表格图片字段配置接口：
  - `/admin/api/tables/<table>/media_fields`
  - `/api/tables/<table>/media_fields`

已存在能力：

- 图片上传
- 图片列表
- 删除前做基础引用检查
- 每张表有可持久化的字段渲染配置

当前不足：

- 删除前引用检查只扫描 docs 和数据库文本列，不扫描前端源码里的硬编码 URL
- 没有媒体绑定表，后端无法精确知道“哪张图被哪个业务对象使用”
- 没有元数据编辑、替换文件、版本化、恢复、批量操作等能力
- 没有历史图片迁移工具


### 2.3 前端现状

当前前端存在三套图片来源：

1. 后台媒体库
2. 业务表中的 `pictureid`
3. 页面源码/文档中的硬编码 URL

当前已知现状：

- 后台已有 `Media` 管理页
- 后台已有“表格图片字段配置”弹窗
- 前台表格图片渲染仍依赖 `src/utils/tableMedia.ts` 里的 legacy 默认规则
- 自然抑制 tRNA 页面和 BLAST 页面仍直接拼接 MinIO 图片 URL
- 首页、导航、bot、About、AI 相关界面仍存在大量硬编码图片地址

当前不足：

- 媒体后台可以“看和传”，但还不能真正“接管”
- 图片字段配置页功能打通了，但还不够适合普通管理员使用
- 页面级图片仍大量绕开媒体库


### 2.4 当前最核心的问题

当前最大的结构性问题不是“缺一个上传页”，而是：

**没有单一图片来源，也没有统一引用模型。**

这会导致：

- 同一张图可能同时出现在媒体库、文档 URL、前端硬编码和业务表字段里
- 无法准确判断删除风险
- 无法知道图片属于谁、被谁使用、是否还能替换
- 后台很难真正成为运营入口

### 2.5 当前代码入口点

为了保证后续改造不偏离当前实现，现阶段图片资源相关的主要代码入口应视为如下几类：

后端入口：

- `Flask/app/models.py`
  - `MediaAsset`
- `Flask/app/routes.py`
  - `/admin/api/media`
  - `/admin/api/media/upload`
  - `/admin/api/media/<id>`
  - `/admin/api/tables/<table>/media_fields`
  - `/api/tables/<table>/media_fields`
- `Flask/app/settings_store.py`
  - `table_media_field_config_json`

前端后台入口：

- `src/views/admin/AdminMediaPanel.vue`
- `src/views/admin/AdminTableMediaFieldsDialog.vue`
- `src/views/admin/AdminWorkspace.vue`
- `src/utils/admin.ts`

前台仍在使用 legacy 图片逻辑的重点入口：

- `src/utils/tableMedia.ts`
- `src/views/natural-sup-tRNA/*`
- `src/views/blast/*`
- `src/views/Home.vue`
- `src/components/NavBar.vue`
- `src/bot/BotComponent.vue`
- `src/views/about/about.vue`

文档图片入口：

- `public/docs/*.md`

后续改造应优先在上述入口收拢逻辑，而不是继续新增新的第四套或第五套图片管理实现。


## 3. 目标

### 3.1 总目标

把 ENSURE 图片资源管理改造成：

**一个可后台运营、可审计、可绑定业务对象、可安全删除、可逐步迁移历史图片的统一资源系统。**


### 3.2 目标结果

改造完成后，图片资源应满足：

- 所有新增图片统一进入媒体资产表
- 表格图片渲染规则不再依赖前端硬编码默认值
- 历史 `pictureid` 图片能逐步纳入媒体库体系
- 文档图片与页面图片逐步纳入统一引用治理
- 删除图片时可以明确知道“谁在引用”
- 管理员可以通过后台完成常见图片运维动作
- 普通管理员可以为业务记录新增“逻辑图片列/图片槽位”，而不需要直接改生产表结构
- 每张图片都可以在后台反查到具体引用位置，至少精确到表/字段/记录或页面/文档槽位
- AI 在回答相关问题时可以获得图片、图片所在位置和图注等上下文，并在必要时进行受约束的多模态理解


### 3.3 非目标

本轮不以以下内容为目标：

- 不做复杂 DAM 级别的图像工作流系统
- 不做视频资源管理
- 不做图片裁剪编辑器
- 不做 CMS 级富文本可视化站点编辑器
- 不在第一阶段就强制清除所有历史 URL
- 不向普通管理员开放任意 `ALTER TABLE`、任意新增物理列的数据库结构修改能力


## 4. 设计原则

### 4.1 单一来源原则

新增图片只能通过媒体库进入系统。

### 4.2 渐进迁移原则

历史图片不能一次性硬切，必须兼容迁移。

### 4.3 渲染与归属分离原则

“字段怎么显示图片”与“图片属于谁”是两件不同的事：

- 渲染规则由表格字段配置决定
- 图片归属与引用关系由媒体绑定关系决定

### 4.4 删除安全优先原则

如果无法证明图片未被引用，则默认不允许删除。

### 4.5 后台优先原则

图片管理必须优先面向普通管理员可用，而不是只对开发者可用。

### 4.6 普通管理员安全边界原则

普通管理员可以配置图片显示、绑定图片、增加逻辑图片位，但不应直接对生产表做任意物理 schema 变更。

换句话说：

- 可以新增“逻辑图片列”
- 不默认允许新增真实 MySQL 列

真实数据库结构变更应保留给：

- 超级管理员
- 受控迁移脚本
- 明确审批后的后台高级功能


## 5. 目标架构

目标状态下，图片资源系统应由三层组成：

1. `media_assets`
2. `media_bindings`
3. `media rendering config`


### 5.1 `media_assets`

作为统一的图片资产主表，负责存储：

- 文件位置
- 基础元数据
- 文件指纹
- 展示信息
- 来源信息

现有 `media_assets` 保留并扩展，不另起新表替代它。


### 5.2 `media_bindings`

新增绑定表，负责回答：

- 这张图属于谁
- 被哪里使用
- 为什么不能删
- 这个页面/记录/字段对应哪张图

推荐结构：

- `id`
- `asset_id`
- `binding_type`
- `resource_name`
- `field_name`
- `record_key`
- `slot_key`
- `extra_json`
- `created_at`
- `updated_at`

字段说明：

- `asset_id`：关联 `media_assets.id`
- `binding_type`：
  - `table_field`
  - `doc_image`
  - `site_asset`
  - `manual_attachment`
- `resource_name`：
  - 表名
  - 文档路径
  - 页面资源名称
- `field_name`：表字段名或文档字段名
- `record_key`：业务记录主键或逻辑标识
- `slot_key`：例如 `site_logo`、`home_figure_1`
- `extra_json`：扩展字段


### 5.3 `table_media_field_config_json`

继续作为“字段渲染规则配置”存储，不负责图片归属。

该配置只描述：

- 这个字段如何显示
- 是否按图片渲染
- URL 如何生成
- 尺寸和预览策略

它不应再承担：

- 图片是否存在
- 图片属于谁
- 图片能不能删


### 5.4 页面/文档图片管理

成熟目标状态下，页面与文档中的图片也应纳入统一管理：

- docs 图片通过 `media_bindings(binding_type='doc_image')`
- 主站资源图通过 `site_asset` 槽位绑定
- 首页 figure、logo、bot avatar、AI 页面头像等不再直接硬编码 URL

### 5.5 AI 图片上下文接入

成熟目标状态下，AI 不应只读取文本证据，也应能够读取与问题相关的图片上下文。

AI 图片上下文应至少包括：

- 图片本身
- `title`
- `alt_text`
- `caption`
- `binding_type`
- `resource_name`
- `field_name`
- `record_key`
- `slot_key`

AI 使用顺序建议：

1. 先使用结构化绑定信息判断图片与问题是否相关
2. 再使用图注、alt text、caption 等文本元数据
3. 只有确有必要时才调用多模态模型读取图片本身

约束：

- 仅凭图片所在位置做出的判断必须视为“推断”，不能伪装成结构化数据库事实
- 基于图像视觉内容的结论应在回答中标注为“根据图片推断”或“根据图注/图片内容判断”

### 5.6 ORM 边界

本项目不应为了图片资源管理而整体改造成 ORM 主驱动系统。

原因：

- 业务数据层本身大量依赖动态表、动态字段、动态过滤
- 当前表查询、统计、导出、AI 工具访问模式不是围绕固定 ORM model 设计的
- 为了图片管理而推动全库 ORM 化，成本远高于收益

因此推荐边界是：

- 业务大表：继续以当前动态表访问为主
- 图片资源子系统：使用稳定 ORM model
- 前端渲染与业务绑定：通过配置和绑定关系衔接

也就是说：

- `media_assets` 继续 ORM 管理
- `media_bindings` 作为新增 ORM 表管理
- 站点资源槽位可以继续先走 `app_settings`，也可在后续演进为稳定小表

不建议：

- 为了图片功能，把所有业务表都建成 ORM model
- 把图片绑定关系塞回业务大表字段中


## 6. 数据模型目标

### 6.1 保留并扩展 `media_assets`

建议在现有基础上补充字段：

- `caption`
- `license`
- `tags_json`
- `status`
- `is_public`
- `updated_at`

可选但有价值的字段：

- `source_ref`
- `entity_type`
- `entity_id`

但如果引入了 `media_bindings`，则 `entity_type / entity_id` 可以不加在主表。


### 6.2 新增 `media_bindings`

建议索引：

- `idx_media_bindings_asset_id`
- `idx_media_bindings_binding_type`
- `idx_media_bindings_resource_name`
- `idx_media_bindings_record_key`
- 唯一约束：
  - `(asset_id, binding_type, resource_name, field_name, record_key, slot_key)`

这样可以：

- 快速查引用
- 防止重复绑定
- 支撑后台引用视图和删除校验


### 6.3 站点级资源配置

站点级静态图片可以先继续放在 `app_settings` 中，用 JSON 管理，不需要第一阶段新建复杂表。

建议新增：

- `site_asset_slots_json`

内容示例：

```json
{
  "site_logo": {"asset_id": 1},
  "bot_avatar": {"asset_id": 2},
  "home_figure_1": {"asset_id": 3},
  "home_figure_2": {"asset_id": 4},
  "about_logo_1": {"asset_id": 5}
}
```

### 6.4 逻辑图片列 / 记录图片槽位

“普通管理员可以新增一列图片列”这个需求是合理的，但应实现为：

- 逻辑图片列
- 记录图片槽位

而不是默认让普通管理员直接新增真实数据库字段。

建议设计：

- 在表配置中增加 `virtual_media_fields`
- 每个逻辑图片列至少包含：
  - `key`
  - `label`
  - `multiple`
  - `placement`
  - `required`
  - `sort_order`

示例：

```json
{
  "virtual_media_fields": [
    {
      "key": "structure_figure",
      "label": "Structure Figure",
      "multiple": false,
      "placement": "record",
      "required": false,
      "sort_order": 10
    }
  ]
}
```

数据保存方式：

- 图片本体存 `media_assets`
- 记录与图片的关系存 `media_bindings`
- `binding_type` 建议新增：
  - `table_record_slot`

这样做的好处：

- 普通管理员可以“加一列图片”
- 但不会直接改坏数据库表结构
- 删除检查和引用追踪仍可统一走绑定关系

### 6.5 绑定关系语义约束

为了保证后续引用反查、AI 图片检索和删除校验可靠，`media_bindings` 的语义应固定：

- 一条 binding 描述“一张图片被一个位置使用”
- 一个位置可以对应多张图，但应通过多条 binding 表达
- 一个图片可以被多个位置复用，但每个位置都必须显式建 binding

建议约束：

- `table_field`
  - 用于真实数据库字段中直接存图片标识或图片 URL 的场景
- `table_record_slot`
  - 用于逻辑图片列/记录图片槽位
- `doc_image`
  - 用于 Markdown 或帮助文档中的图片
- `site_asset`
  - 用于站点固定资源位
- `manual_attachment`
  - 用于临时或人工绑定但未正式归类的情况

推荐 `record_key` 规则：

- 如果表有主键，用主键值
- 如果表无单主键，用稳定逻辑键，例如 `ENSURE_ID` 或复合标识序列化

推荐 `slot_key` 规则：

- 站点资源：`site_logo`、`home_figure_1`
- 逻辑图片列：使用 `virtual_media_fields.key`
- 文档图片：可使用 `section_id` 或图片序号


## 7. 后台目标能力

### 7.1 媒体库管理页目标

管理员应能够：

- 搜索图片
- 筛选图片
- 上传图片
- 查看大图和详情
- 编辑元数据
- 查看引用
- 复制 URL / Markdown
- 安全删除未引用图片

必须支持：

- 列表
- 上传
- 详情
- 删除阻止
- 引用视图
- 元数据编辑


### 7.2 字段配置页目标

管理员应能够：

- 找到疑似图片字段
- 配置字段渲染方式
- 即时预览效果
- 保存配置
- 明确知道哪些字段改了

目标体验：

- 不默认把所有字段全部铺开
- 应优先显示候选字段
- 应提供模板预设
- 应自动选一个非空样例进行预览
- 应有字段级校验

### 7.3 记录编辑页中的图片槽位目标

在后台编辑单条记录时，普通管理员应能够：

- 为当前表启用逻辑图片列
- 为当前记录上传或绑定一张/多张图片
- 明确看到该图片位名称
- 明确知道当前记录已经绑定了哪些图片
- 替换图片或移除图片

### 7.4 真实图片列的直接编辑目标

除了逻辑图片列/记录图片槽位外，后台还必须支持一种更贴近普通管理员直觉的能力：

- 当某个真实字段本质上是图片字段时，管理员在“编辑记录”时看到的应是图片，而不是原始 `pictureid`

这条能力首先应用于：

- 已确认的历史 `pictureid` 字段
- 已配置为 `renderer = image` 的真实表字段

目标交互：

1. 在行编辑弹窗中，如果字段被识别为图片字段：
   - 默认展示图片预览卡片
   - 展示当前绑定状态
   - 展示“更换图片 / 从媒体库选择 / 移除图片”操作
2. 不默认暴露原始 `pictureid` 文本输入
3. 仅在“高级模式”或调试模式下允许查看/修改底层原始值

推荐交互结构：

- 字段标签
- 当前图片缩略图 / 空状态
- 当前来源说明
  - 绑定关系
  - legacy `pictureid`
- 操作按钮：
  - 选择图片
  - 上传并替换
  - 清除图片
  - 查看引用
- 折叠区：
  - 原始 `pictureid`
  - 只对高级管理员可见

底层约束：

- 对真实 `pictureid` 字段：
  - UI 层按图片编辑
  - 存储层仍允许保留原字段值
  - 同时应建立或更新 `media_bindings(binding_type='table_field')`
- 对逻辑图片列：
  - UI 层按图片编辑
  - 不写真实物理列
  - 只更新 `media_bindings(binding_type='table_record_slot')`

这样可以同时满足：

- 管理员体验上“我在改图片”
- 系统层面仍然保持兼容与可回滚

不应接受的做法：

- 在真实图片字段上继续只显示 `pictureid`
- 让普通管理员自己手工输入 MinIO URL
- 让普通管理员自己记忆 `pictureid` 规则

### 7.5 行编辑器图片字段的实施顺序

这项能力不应一次性套到所有字段，而应分两步落地：

#### 第一步：legacy `pictureid` 字段

先只对这些已确认表启用：

- `nonsense_sup_rna.pictureid`
- `frameshift_sup_trna.pictureid`

交互目标：

- 列表里可直接看到图片缩略图
- 行编辑弹窗里直接编辑图片，而不是编辑 `pictureid`
- 仍保留旧 `pictureid` 作为兼容兜底

#### 第二步：后台配置为图片字段的真实列

当 `media_fields` 中字段被标记为 `renderer = image` 后：

- 行编辑器自动切换为图片编辑器
- 不再默认使用普通文本输入框

#### 第三步：逻辑图片列/记录图片槽位

当表启用了 `virtual_media_fields` 后：

- 行编辑器中单独出现“图片槽位”区域
- 支持多图绑定和排序（如后续需要）

不应要求管理员：

- 手工写 URL
- 手工拼接 MinIO 路径
- 手工新增数据库真实列

### 7.4 图片引用反查目标

每张图片在后台详情页都应能看到完整引用列表。

最少应展示：

- 引用类型
  - `table_field`
  - `table_record_slot`
  - `doc_image`
  - `site_asset`
- 表名 / 页面名 / 文档名
- 字段名或槽位名
- 记录主键或业务标识
- 引用创建时间

如果是业务表引用，应尽量支持：

- 跳转到该表
- 打开对应记录
- 高亮对应字段或图片槽位

### 7.5 站点资源管理目标

后台后续应支持管理：

- site logo
- bot avatar
- homepage figures
- about logos
- AI 页面头像等

管理员应该通过“从媒体库选图”完成配置，而不是手写 URL。

### 7.6 AI 图片管理相关后台目标

后台后续应支持为图片补充 AI 友好的描述信息：

- `alt_text`
- `caption`
- 业务说明
- 图片归属位置

这样 AI 在回答时才能优先读取结构化上下文，而不是只依赖视觉推断。

### 7.7 后台典型用户流程

为避免后续改造只顾功能、不顾使用路径，后台应至少满足以下流程闭环。

#### 流程 A：上传一张新图并立即绑定到某条记录

1. 管理员进入 `Media`
2. 上传图片并填写：
   - title
   - alt_text
   - caption
   - source_type
3. 上传成功后可直接点击“绑定到记录”
4. 选择：
   - 表
   - 记录
   - 字段或逻辑图片槽位
5. 保存 binding
6. 回到记录编辑页可见图片已生效

#### 流程 B：为某张表新增一个逻辑图片列

1. 管理员进入某张表配置
2. 打开“图片字段/图片槽位”配置
3. 新增 `virtual_media_field`
4. 设置：
   - key
   - label
   - multiple
   - required
   - sort_order
5. 保存后，后台记录编辑页出现该图片槽位
6. 管理员可为任意记录上传或绑定图片

#### 流程 C：查看某张图片被哪里使用

1. 管理员进入 `Media`
2. 点击某张图片详情
3. 在引用区看到：
   - 业务表引用
   - 文档引用
   - 页面槽位引用
4. 可跳转到对应表记录或页面资源配置

#### 流程 D：尝试删除图片

1. 管理员在媒体详情点击删除
2. 系统先做 binding 级检查
3. 若存在引用：
   - 直接阻止删除
   - 返回引用列表
4. 若无引用：
   - 允许删除
   - 记录审计


## 8. 删除与引用治理目标

### 8.1 删除规则

删除图片前必须检查以下来源：

- `media_bindings`
- docs 引用
- 数据库字段引用
- 站点资源槽位引用
- 兼容期的 legacy 字段映射引用

如果任一引用存在：

- 默认严格禁止删除
- 返回引用列表
- 记录审计日志


### 8.2 删除校验目标

目标状态下不再依赖“全文扫描数据库文本列”作为唯一判断依据。

全文扫描只应作为过渡期补充手段。

真正的删除治理应以结构化绑定关系为主。

### 8.3 删除校验执行顺序

后续删除校验建议按以下顺序执行：

1. 先查 `media_bindings`
2. 再查站点资源槽位配置
3. 再查 docs 引用索引
4. 再查 legacy 映射关系
5. 最后才做全文扫描兜底

这样做的原因：

- 结构化关系最可靠
- 全文扫描最慢，也最容易误判

只有当前 1-4 都无法覆盖的历史阶段，才允许依赖第 5 步。

### 8.4 删除返回格式目标

删除被阻止时，后端返回应统一为结构化 JSON，而不是只返回一条字符串错误。

推荐格式：

```json
{
  "ok": false,
  "code": "MEDIA_ASSET_IN_USE",
  "message": "Asset is still referenced.",
  "references": [
    {
      "binding_type": "table_record_slot",
      "resource_name": "nonsense_sup_rna",
      "field_name": "",
      "record_key": "123",
      "slot_key": "structure_figure"
    }
  ]
}
```

这样前端才能真正做出“引用详情视图”，而不是只弹一条错误提示。


## 9. 前端接入目标

### 9.1 表格页面

所有表格类图片显示逻辑应优先读取后台配置：

1. 后台 `media_fields`
2. 结构化绑定/映射
3. legacy 规则兜底

最终目标是逐步去掉 legacy 默认规则。


### 9.2 文档页

文档图片不应长期继续纯手写外链 URL。

中期目标：

- 文档图片能引用媒体库中的图片
- 删除时能识别文档引用


### 9.3 页面硬编码图片

首页、导航、About、bot、AI 页面等硬编码图片必须逐步迁到后台管理。

迁移顺序建议：

1. logo
2. bot avatar
3. homepage figures
4. about logos
5. AI 页面装饰图

### 9.4 AI 图片接入目标

AI 在以下场景应可获得相关图片上下文：

- 用户询问某条记录对应图片
- 用户询问某张结构图、示意图、logo、figure 的含义
- 用户要求根据页面 figure 或图注解释某个数据库模块
- 用户让 AI 对图片进行辅助说明

AI 检索顺序建议：

1. 根据问题先检索结构化绑定关系
2. 获取图片对应的标题、图注、alt text、所属位置
3. 如果这些信息仍不足，再读取图片 URL 或调用多模态模型

回答约束：

- 来自绑定位置的判断，必须标注为“根据图片位置或绑定关系推断”
- 来自图注的判断，可视作较强文本证据
- 来自图片本身视觉内容的判断，应标注为“根据图片内容判断”
- 不能把图片推断直接冒充成数据库字段事实

### 9.5 前端渲染回退顺序

后续所有前端图片渲染，应遵循一致的回退顺序，避免继续出现每个页面各写一套逻辑。

推荐顺序：

1. 先查逻辑图片槽位绑定
2. 再查真实字段对应的 `media_bindings`
3. 再查后台 `media_fields` 配置
4. 最后才允许 legacy `pictureid -> MinIO URL` 规则兜底

禁止新增：

- 新页面继续手工拼接图片 URL
- 新组件继续直接读取 `pictureid` 后自己拼图

### 9.6 图片上下文提供给 AI 的数据包

AI 侧后续不应只拿一个图片 URL，而应拿一个统一上下文对象。

推荐结构：

```json
{
  "asset_id": 12,
  "public_url": "https://...",
  "title": "Structure figure",
  "alt_text": "Predicted suppressor tRNA structure",
  "caption": "Secondary structure of ...",
  "binding_type": "table_record_slot",
  "resource_name": "nonsense_sup_rna",
  "field_name": "",
  "record_key": "123",
  "slot_key": "structure_figure"
}
```

这样 AI 才能区分：

- 这是一张什么图
- 它出现在什么位置
- 这是结构化归属，还是仅仅一张散图


## 10. 迁移策略

### 10.1 P1：媒体库后台可运营化

目标：

- 媒体页可真正用于新增图片管理

内容：

- 列表
- 上传
- 删除阻止
- 详情
- 元数据编辑
- 引用视图

状态：

- 当前已完成基础版，但仍需补强普通管理员体验


### 10.2 P2：表格图片字段配置后台化

目标：

- 替换前端硬编码 `tableMedia.ts` 规则作为主配置来源

内容：

- 后台字段渲染配置
- 预览
- 保存
- 优先使用后台配置

状态：

- 当前已打通基础版，但尚未达到“足够方便用户使用”的程度


### 10.3 P3：历史图片迁移

目标：

- 把历史 `pictureid` 与文档图片逐步纳入媒体库和绑定关系

内容：

- 先做后端预览与迁移入口
- 扫描表中的历史图片字段
- 为历史图片生成或复用 `media_assets`
- 写入 `media_bindings`
- 建兼容映射
- 在迁移完成且核验通过前，不切公网页面图源

原则：

- 先建绑定，再改读取逻辑
- 不直接硬改所有历史字段

#### 10.3.1 P3 的分步执行

P3 必须拆成两步，而不是一次性切换：

##### P3-A：后端迁移支撑

目标：

- 只建立历史图片的结构化关系
- 不改前端实际显示图源

内容：

- 增加迁移预览接口
- 增加迁移执行接口
- 扫描 `pictureid` 非空的业务表
- 为每个历史图片生成或复用 `media_assets`
- 为每条记录写入 `media_bindings(binding_type='table_field')`
- 在 `extra_json` 中保留历史字段上下文，例如：
  - `legacy_pictureid`
  - `legacy_url`
  - `source_field`
  - 可用图注候选

说明：

- 这一阶段完成后，后台能看到历史图片引用关系
- 但前台页面仍继续使用原来的 `pictureid -> URL` 逻辑
- 这一阶段的目标是“建立关系”，不是“改显示”

##### P3-B：读取链路切换

目标：

- 让前端显示优先读取结构化绑定

内容：

- 表格/页面图片渲染优先读：
  1. `table_record_slot`
  2. `table_field`
  3. legacy `pictureid`
- 对重点页面逐步切换，而不是全站硬切

说明：

- 只有在 P3-A 数据核验通过后，才允许进入 P3-B
- P3-B 期间仍保留 legacy 兜底

#### 10.3.2 P3-A 的迁移对象

第一批迁移对象仅限历史 `pictureid` 字段，不包括 docs 和站点硬编码图片。

迁移范围：

- 所有可管理业务表中，存在 `pictureid` 列且值非空的记录

当前已确认的首批重点表：

- `nonsense_sup_rna`
- `frameshift_sup_trna`

当前数据库实测迁移盘点：

- 候选表数量：`2`
- 候选记录数量：`86`
- 可稳定生成 `record_key` 的数量：`86`
- 当前迁移前提状态：
  - `migration_ready = true`
  - `bucket_configured = true`
  - `public_base_configured = true`

当前分表候选数量：

- `nonsense_sup_rna`
  - `pictureid` 非空记录：`52`
- `frameshift_sup_trna`
  - `pictureid` 非空记录：`34`

这意味着第一批历史图片迁移可以直接按这 `86` 条记录执行，不需要再等待额外表结构准备。

第一批暂不处理：

- docs Markdown 图片
- 首页/导航/bot/about 等硬编码图片
- 非 `pictureid` 模式的历史图片字段

#### 10.3.2.1 P3-A 的执行清单

在真正按批次执行迁移时，应严格按下面顺序：

1. 先跑 `preview`
   - 确认候选表
   - 确认候选记录数
   - 确认每条记录都能生成稳定 `record_key`
2. 跑 `dry_run`
   - 确认预计新增多少 `media_assets`
   - 确认预计新增多少 `media_bindings`
   - 确认是否存在复用同一图片资源的情况
3. 执行数据库备份
4. 仅对首批表执行正式迁移
   - 第一批：`nonsense_sup_rna`
   - 第二批：`frameshift_sup_trna`
5. 在后台媒体详情页抽样核验：
   - 是否能看到引用反查
   - 是否能看到表名 / 字段名 / 记录键
6. 不改前台读图逻辑
7. 如果核验通过，再进入 P3-B

不建议的执行方式：

- 一次性把 docs 和站点硬编码图片混入同一批迁移
- 不做 dry-run 直接正式写入
- 在没有抽样核验前就切前台读图优先级

#### 10.3.3 P3-A 的部署策略

P3-A 应按“后端优先、前端不动”执行：

1. 部署后端
2. 自动创建/校验新增表结构
3. 调用迁移预览接口，确认候选数量
4. 执行迁移写入
5. 在后台媒体详情里核验引用结果
6. 若异常，先回滚迁移写入，不发布前端

P3-A 结论：

- 只需要重部署后端
- 不需要同步发布前端
- 不需要切 nginx 规则

#### 10.3.4 P3-B 的部署策略

P3-B 才会影响公网页面图片读取优先级，因此必须：

1. 先完成 P3-A
2. 再部署后端兼容逻辑
3. 再部署前端读取逻辑
4. 用重点页面做人工回归

P3-B 结论：

- 需要重部署前后端
- 仍不应删除 legacy `pictureid` 兜底

#### 10.3.5 P3 的成功标准

P3-A 成功的标志：

- 历史 `pictureid` 可以预览出待迁移记录数
- 迁移后 `media_assets` 中出现对应资产
- 迁移后 `media_bindings` 中出现对应 `table_field` 绑定
- 媒体详情页能反查出表名、字段名、记录键
- 旧页面显示不受影响

P3-B 成功的标志：

- 新渲染链路能优先读绑定关系
- 读不到绑定时仍能回落到 legacy `pictureid`
- 不出现掉图
- 删除检查能识别历史迁移后的结构化引用

#### 10.3.6 P3 与行编辑器改造的关系

“编辑记录时直接编辑图片，而不是编辑 `pictureid`” 应当作为 P3 的一部分协同推进。

推荐顺序：

1. 先完成 P3-A
   - 确保历史 `pictureid` 已经有结构化绑定
2. 再改行编辑器
   - 对 `pictureid` 字段显示图片编辑控件
   - 通过媒体库选择/上传来完成变更
3. 再进入 P3-B
   - 让前台读取优先级切到绑定关系

这样做的原因：

- 如果先改行编辑器，但还没有绑定关系，后台仍然会回落到编辑原始 ID，体验并不会真正改善
- 先有绑定关系，行编辑器才能真正以图片为中心，而不是以字符串 ID 为中心

#### 10.3.7 行编辑器图片字段的成功标准

以下条件满足后，才算“图片列可直接编辑图片”真正落地：

- 管理员在行编辑弹窗中看到的是图片预览，而不是 `pictureid`
- 管理员可直接从媒体库选择图片替换当前图片
- 替换后：
  - 绑定关系正确更新
  - 如为真实 `pictureid` 字段，底层兼容值也同步正确
- 清除图片后：
  - 绑定关系被移除
  - 前台显示仍符合回退规则
- 普通管理员不需要理解 `pictureid` 的具体命名规则


### 10.4 P4：站点图片统一接管

目标：

- 让核心页面图片进入后台可管理体系

内容：

- 站点资源槽位管理
- 页面接入媒体库
- 去除高价值硬编码图片

### 10.5 P4.5：AI 图片上下文接入

目标：

- 让 AI 可以在受控前提下获取与问题相关的图片资源

内容：

- 提供图片绑定查询能力
- 提供图片文本元数据查询能力
- 必要时提供图片 URL 或多模态读取能力
- 在 AI 侧区分“数据库事实”和“图片推断”

### 10.6 P5：移除 legacy 依赖

目标：

- 最终下线前端默认硬编码规则和旧拼接逻辑

内容：

- 清理 `tableMedia.ts` 旧规则
- 清理页面内直拼 MinIO URL 逻辑
- 删除不再需要的过渡兼容代码

### 10.7 每阶段交付物

为了让后续执行更可控，每个阶段都应有明确交付物，而不是只写“做完某个能力”。

#### P1 交付物

- 媒体后台页
- 上传、列表、详情、删除阻止
- 元数据编辑接口和前端
- 引用详情占位能力

#### P2 交付物

- 表格图片字段配置页
- 候选字段模式
- 模板预设
- 非空样例预览
- 字段级校验

#### P3 交付物

- `media_bindings` 表
- 历史图片迁移脚本
- 迁移日志
- 兼容映射读取逻辑

#### P4 交付物

- 站点图片槽位配置
- 首页/导航/bot 等页面接入
- 删除检查支持站点槽位引用

#### P4.5 交付物

- AI 图片上下文查询接口或内部能力
- AI 使用图片上下文的规则
- 多模态启用边界

#### P5 交付物

- legacy 代码清单
- 清理 PR 或提交
- 回归验证报告


## 11. 兼容策略

整个迁移过程中应遵循：

- 新逻辑优先
- 旧逻辑兜底
- 先双轨，后清理

即：

- 新上传图片立即进入媒体库
- 历史图片先做映射
- 页面优先读绑定关系或新配置
- 读不到时才回落到旧 `pictureid`


## 12. 管理员可用性目标

当前后台功能“能用”，但还不够“顺手”。

目标状态必须满足：

- 普通管理员不需要自己发明 URL 模板
- 能快速找到疑似图片字段
- 能看到明确预览
- 能知道哪些图片正在被用
- 删除行为风险可解释

必须补的体验项：

1. 候选字段模式
2. 模板预设
3. 自动选非空样例
4. 字段级校验
5. 引用列表
6. 元数据编辑


## 13. 审计与安全目标

所有图片资源相关操作必须记录审计：

- 上传
- 删除
- 编辑元数据
- 配置字段渲染
- 绑定/解绑图片
- 修改站点槽位图片

建议审计字段至少包含：

- operator
- action
- resource_type
- resource_id
- before_json
- after_json
- created_at

### 13.1 权限边界目标

图片资源管理至少应区分三类权限：

1. 普通管理员
   - 上传图片
   - 编辑元数据
   - 查看引用
   - 配置字段渲染
   - 配置逻辑图片列
   - 为记录绑定/解绑图片

2. 高级管理员
   - 站点图片槽位管理
   - 历史图片迁移执行
   - 批量修复绑定关系

3. 超级管理员
   - 受控物理 schema 变更
   - 强制解绑或危险操作
   - 清理 legacy 兼容逻辑

如果当前项目短期内不做复杂权限体系，至少也应在文档中明确这三种能力边界，避免普通管理员默认获得过高权限。


## 14. 验收标准

### 14.1 第一阶段验收

- 后台能查看和上传图片
- 后台能阻止删除已引用图片
- 后台能配置表格图片字段
- 前端表格开始优先读后台配置
- 后台可以为表启用逻辑图片列/图片槽位，而不需要直接新增真实数据库列


### 14.2 第二阶段验收

- `pictureid` 历史图片能映射进媒体库
- 删除检查能识别绑定关系
- docs 图片可纳入媒体引用管理
- 每张图片都能在后台看到精确引用位置


### 14.3 第三阶段验收

- 首页、导航、bot 等核心图片可通过后台替换
- 页面不再依赖手写 URL
- AI 可获得图片位置、图注和相关图片元数据


### 14.4 最终验收

- 新增图片统一进入媒体库
- 图片引用均可追踪
- 删除安全可靠
- 业务表图片不再依赖前端硬编码默认规则
- 高价值页面图片可后台管理

### 14.5 测试矩阵

后续每一阶段改造都至少应覆盖以下测试维度：

#### 后端

- 上传图片成功
- 重复文件处理正确
- 删除未引用图片成功
- 删除已引用图片被阻止
- 引用列表返回结构正确
- 逻辑图片槽位绑定成功
- AI 图片上下文查询返回正确

#### 前端后台

- 媒体列表可搜索和筛选
- 图片详情可见元数据
- 记录编辑页能绑定/解绑图片
- 字段配置页能保存和回显
- 候选字段模式工作正常

#### 前台

- 表格页图片显示正确
- 历史 `pictureid` 兼容正常
- 首页/导航/bot 图片接入正常
- docs 图片不丢失

#### AI

- 能根据图片所在记录找到图片
- 能优先使用图注/alt_text
- 能在需要时说明“这是根据图片推断”
- 不会把图片推断说成数据库字段事实


## 15. 实施顺序建议

建议严格按以下顺序推进：

1. 先补强 P2 的管理员可用性
2. 再做 `media_bindings`
3. 再做历史图片迁移
4. 再接管站点硬编码图片
5. 再接入 AI 图片上下文能力
6. 最后清理 legacy 逻辑

不建议：

- 直接全删 `pictureid` 逻辑
- 在没有绑定表时就把删除判断完全交给媒体库
- 未迁移历史图片前就强制全站只读媒体库

### 15.1 回滚策略

为了保证后续改造可控，每一阶段都必须可以回滚。

建议回滚策略：

- P1/P2
  - 前端回退到上一版后台界面
  - 保留现有 `table_media_field_config_json`
- P3
  - 迁移脚本必须输出映射日志
  - `media_bindings` 写入应可按批次回滚
  - 旧 `pictureid` 逻辑在确认稳定前不得删除
- P4/P4.5
  - 页面图片改造先做双轨读取
  - AI 图片上下文接入先做可关闭开关
- P5
  - 只有在完整回归通过后，才允许删 legacy 逻辑


## 16. 当前结论

我对这套改造是有信心的，但前提是：

- 按阶段渐进迁移
- 不做一次性大爆改
